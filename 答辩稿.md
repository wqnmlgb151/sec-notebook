# sec-notebook — Web 安全渗透测试学习平台

> **Web前端应用开发** 课程结课大作业 答辩材料

---

## 一、我做了一个什么项目

我做了一个叫 **sec-notebook** 的网站，主题是 Web 安全知识学习。

简单说就是：**一个介绍常见网站漏洞有哪些、怎么攻击、怎么防御的在线学习平台。**

打开网站能看到 5 个页面：首页、学习笔记、漏洞详情、安全实验室、关于项目。手机、平板、电脑都能正常看。

---

## 二、用到了哪些技术

这个项目没有用后端（没有 Python/Java/数据库那一套），是一个纯前端的静态网站。

| 技术 | 干什么用的 |
|------|-----------|
| HTML | 页面的骨架——标题、按钮、输入框这些元素的定义 |
| CSS | 页面的皮肤——颜色、布局、动画效果 |
| JavaScript | 页面的行为——点击按钮发生什么、表单怎么检查 |
| Vue 3 | 一个简化开发的框架——让数据和页面自动同步，不用手动操作 |

换句话说：HTML 决定"有什么"，CSS 决定"长什么样"，JavaScript 决定"做什么"，Vue 让数据和界面自动联动。

别人做类似项目可能需要装一堆工具（Node.js、npm、Webpack），我这个项目不需要——所有需要的库都通过 CDN（网上别人托管好的链接）直接引入，双击 HTML 文件就能在浏览器里跑起来。

## 三、5 个页面分别是什么

### 3.1 首页

打开网站看到的第一个页面。上面有项目的标题（带了一个故障抖动的动画效果），中间用卡片展示了 5 个安全知识模块，下面列出了全部 19 篇学习笔记的入口。配色是暗黑色系 + 青色发光，和"黑客/安全"的主题比较搭。

### 3.2 学习笔记（最核心的页面）

这个页面会自动从我的 GitHub 仓库（一个存笔记的网盘）把 19 篇 Markdown 笔记抓下来，显示在左边的目录里。点左边任意一篇，右边就会把 Markdown 内容渲染成漂亮的网页格式——有代码高亮、表格、标题层级。

这里做了几个细节处理：
- **加载中**：显示灰色骨架屏占位（灰色条在闪烁），告诉用户在加载
- **加载失败**：显示重试按钮，不会白屏卡死
- **API 挂了**：GitHub 有时候会限流不让抓，代码会自动切换到内置的备用列表

### 3.3 漏洞详情

深度讲解了 6 种常见的 Web 漏洞：
1. **SQL 注入** — 在输入框里塞 SQL 语句去偷数据库
2. **XSS 跨站脚本** — 在网页里注入恶意 JavaScript 代码
3. **CSRF** — 伪造用户请求，让用户在不知情的情况下执行操作
4. **SSRF** — 利用服务器去访问内网的机器
5. **文件上传漏洞** — 上传恶意文件到服务器
6. **IDOR** — 通过改 URL 里的 ID 去访问别人的数据

每个漏洞都按同样的结构来讲：**是什么原理 → 怎么攻击（有代码示例）→ 怎么防御**。

### 3.4 安全实验室

一个登录和注册的表单页面。虽然不会真的把数据发给服务器，但前端该做的验证都做了：
- 邮箱格式对不对
- 手机号是不是 11 位
- 密码够不够强（弱/中/强/非常强四个等级实时显示）
- 两次密码输得一不一样

介绍了项目的背景、时间线和用到的技术。

## 四、页面长什么样（设计）

整体风格选了**暗黑色系**，类似代码编辑器或终端的配色，跟"安全/黑客"主题协调。不是那种白底蓝按钮的常见模板。

主要做了几个视觉效果：
- 卡片是半透明的毛玻璃质感
- 重要文字有青色或紫色的发光效果
- 整页有一层很淡的扫描线叠加（不仔细看注意不到）
- 标题鼠标移上去会抖动错位一下

布局上用了一个叫 Grid 的技术，让卡片能自动适应屏幕宽度——手机上单列、平板上两列、电脑上三列或更多。

## 五、遇到的困难和怎么解决的

### 困难 1：网站是静态的，但笔记要能更新

正常的做法是搞个后端服务器 + 数据库。但课程要求是纯前端项目，我就换了个思路：让网页启动时自动去 GitHub 抓我的笔记文件。GitHub 免费提供了一个接口（API），可以读取仓库里的文件列表和内容。这样我在 GitHub 上更新了笔记，网站自动就同步了，不需要重新部署。

如果 GitHub 接口挂了（免费接口有次数限制），代码会自动切到一份备份列表，保证页面还能用。

### 困难 2：渲染 Markdown 时的安全问题

Markdown 转成 HTML 后，如果直接插到页面上，可能会有恶意脚本被执行（XSS 攻击）。我的处理方式是：在插入页面前先用一个叫 DOMPurify 的工具把 HTML "洗一遍"，过滤掉危险的标签和属性。同时在页面里配置了 CSP（内容安全策略），限制了哪些来源的脚本可以执行。

### 困难 3：快速点笔记时的乱序问题

用户快速点左边目录切换笔记时，如果前一个请求还没返回、后一个已经返回了，页面会先显示新的、再被旧的结果覆盖，出现内容闪烁。

解决方式：每次发新请求前，先取消上一个还没完成的请求（用了一个叫 AbortController 的机制）。这样旧请求的结果回来时会被直接丢弃，不会干扰当前显示。

## 附：老师可能追问的底层原理（提前准备）

### Q1：Vue 3 是怎么做到"数据变了页面自动更新"的？

你可以这样回答：

Vue 用了**响应式系统**。简单说就是：我把数据放在 Vue 的 `ref()` 或 `reactive()` 函数里包一层，Vue 会跟踪哪些地方用到了这个数据。当数据变化时，Vue 自动通知用到它的地方去更新。

打个比方——就像 C 语言里的指针：你改了一个指针指向的值，所有引用了这个地址的地方都会读到新值。Vue 做的是类似的事，只是它帮你自动完成了"通知更新"这一步，你不用手动去调刷新函数。

### Q2：v-html 渲染 Markdown 为什么有 XSS 风险？

你可以这样回答：

`v-html` 是 Vue 提供的一个指令，作用是把一段字符串直接当 HTML 插到页面里——浏览器会解析并执行其中的标签和脚本。

这就像 C 语言里直接用 `strcpy` 不做边界检查一样危险：如果字符串里藏了 `<script>alert('攻击')</script>`，浏览器就会执行它。这就是 XSS（跨站脚本攻击）。

我的防护是：在插入页面前先用 DOMPurify 这个库把字符串"洗一遍"，把 `<script>`、`onclick` 这些危险的东西过滤掉，只保留安全的标签（比如 `<p>`、`<h1>`、`<table>`）。

### Q3：GitHub API 抓数据具体怎么实现的？

你可以这样回答：

GitHub 提供了一个 REST API，就是一个 HTTP 地址，访问它能拿到仓库文件的 JSON 列表。我用 JavaScript 的 `fetch()` 函数（类似于 C 语言的网络请求库）去请求这个地址，拿到返回的 JSON 后解析出文件名列表，再挨个显示出来。

流程图：`浏览器 → 发 HTTP 请求 → GitHub 服务器 → 返回 JSON → JavaScript 解析 → 渲染到页面`

如果请求失败（比如 GitHub 限制访问频率），代码会 catch 到错误，然后切换到一个写死的 19 篇文件列表作为备用。

### Q4：响应式布局怎么做到手机/平板/电脑自动适配？

你可以这样回答：

CSS 里有一个叫**媒体查询**（Media Query）的机制，可以检测屏幕宽度然后应用不同的样式：

```css
@media (max-width: 768px)  { /* 手机样式 */ }
@media (min-width: 769px)  { /* 平板样式 */ }
@media (min-width: 1025px) { /* 桌面样式 */ }
```

就像 C 语言里的 `#ifdef` 条件编译——不同条件下编译不同的代码。这里不同屏幕宽度下应用不同的 CSS 规则。

另外我用的是 CSS Grid 和 Flexbox 这两种布局方式，它们天生就能根据可用空间自动调整元素的排列和大小，不用写死像素值。

### Q5：AbortController 是怎么取消请求的？

你可以这样回答：

`fetch()` 发出去的网络请求是可以取消的——只要在发请求时传一个 `signal` 参数，想取消时调用 `abort()` 方法就行了。

类比 C 语言：就像你开了个线程去做网络请求，如果需要取消，调用 `pthread_cancel()` 把这个线程停掉。`AbortController` 就是 JavaScript 里做这件事的工具。

我的场景是：用户快速点左边目录时，前一个请求可能还没返回。不取消的话，新旧结果互相覆盖，页面会闪烁。所以每次新请求前先 `abort()` 掉上一个。

### Q6：Markdown 是怎么变成网页的？

你可以这样回答：

**两步走**：
1. **解析**：用一个库（marked.js）把 Markdown 文本解析成 HTML 字符串。比如 `**粗体**` 变成 `<strong>粗体</strong>`。
2. **高亮**：再用另一个库（highlight.js）找到代码块，按语言（Python、SQL 等）加上颜色标注。

类比：就像编译器——marked.js 是"前端"，把 .md 语法树转成 HTML；highlight.js 是"后端"，负责代码美化。

### Q7：为什么用 CDN 而不是 npm 安装？

你可以这样回答：

npm 是 Node.js 的包管理工具，类似 C 语言的包管理器。用 npm 的话需要装 Node.js、跑 `npm install`、配 Webpack 或 Vite 来打包——整个工具链很重。

CDN 就是别人把库文件托管在公网服务器上，我直接在 HTML 里写一行 `<script src="网址"></script>` 就能用。省去了所有构建步骤，项目的复杂度大大降低。

代价是依赖外部服务——如果 CDN 挂了库就加载不了。更正规的项目应该自建或使用 npm + 构建工具。

### Q8：CSP（内容安全策略）是怎么防 XSS 的？

你可以这样回答：

CSP 是浏览器提供的一个安全机制——在 HTML 的 `<meta>` 标签里声明一个白名单，告诉浏览器"只允许从这些来源加载脚本/样式/图片"。

比如我设了 `script-src 'self' https://unpkg.com`，意思是：只有我自己网站的脚本和 unpkg 这个 CDN 的脚本可以执行。其他来源的——包括被 XSS 注入的恶意 `<script>`——浏览器统统拒绝。

类比 C 语言：像一个内存访问白名单，只允许访问指定的安全区域，其他地址访问直接拒绝。这是在浏览器层面做的最后一道防线。
